## COE 608 Lab 4 – Data Memory (VHDL) Project Overview

This lab implements and tests a **32-bit data memory** in VHDL for a simple processor datapath. The core design is a synchronous memory block with:

- **256 words** of **32-bit** data (`0` to `255` address range).
- Separate **enable** and **write-enable** signals to control access.
- A **clocked process** that performs memory **read** or **write** on a specific clock edge.

The Quartus project files (`4a.qpf`, `4a.qsf`, `4a.qws`) and simulation assets (`data_mem.vwf`, `db`, `incremental_db`, `output_files`, `simulation/`) are there to build, program, and verify the design using Intel Quartus II and its waveform/simulator tools.

---

## Files and Their Roles

- **`4a.vhd`** / **`data_mem.vhd`**  
  These define the main **VHDL entity and architecture** for the data memory:
  - `ENTITY data_mem` declares the interface (ports).
  - `ARCHITECTURE Description` implements the behavior using a clocked process and an internal RAM array.
  - Functionally, `4a.vhd` and `data_mem.vhd` contain the same logic (alternate copies/versions of the same memory component).

- **`4a.qpf`, `4a.qsf`, `4a.qws`**  
  Quartus project and settings files:
  - **`.qpf`** – Quartus Project File (top-level project descriptor).
  - **`.qsf`** – Quartus Settings File (device, pin assignments, compile options).
  - **`.qws`** – Quartus workspace/session info (IDE layout, open files, etc.).
  Together, they let Quartus recognize this directory as the **Lab 4 project** and know which VHDL file is top-level and which FPGA/device to target.

- **`data_mem.vwf`, `data_mem.vwf.temp`**  
  Quartus waveform files:
  - Define **test waveforms** for signals like `clk`, `addr`, `data_in`, `wen`, `en`.
  - Used by the **functional/timing simulator** to apply test vectors and observe `data_out`.
  - The `.temp` variant is an intermediate/backup file generated by the tool.

- **`4b/` directory**  
  Contains additional files for part 4b of the lab (often extended tests, alternate configurations, or additional components). These build on the same memory concept but may integrate with a larger datapath or different test setups.

- **`db/`, `incremental_db/`**  
  Quartus internal compilation databases:
  - Store intermediate results (netlists, technology mapping, timing) from previous compilations.
  - Speed up **incremental recompiles** by reusing prior compile state.
  - Not hand-edited; managed by Quartus.

- **`output_files/`**  
  Build artifacts produced by Quartus compilation:
  - Bitstream/programming files for the FPGA.
  - Reports and logs related to the final implementation.

- **`simulation/` (including `simulation/qsim`)**  
  Simulation-related directories and files:
  - Contain configuration for Quartus’s built-in simulator (`qsim`) and possibly generated simulation models.
  - Used to run and inspect waveforms verifying read/write behavior over time.

## LDA Datapath and Waveforms (High‑Level Summary)

For the CPU datapath portion of the course, the **LDA** instruction is documented in more detail in `LDA_datapath.md`. At a high level:

- **Goal of LDA in this lab**:
  - Copy one 32‑bit word from **data memory** into register `A`, using the **16‑bit address field** in the low bits of the instruction stored in `IR`.
  - Formally: `A <= M[ IR[15..0] ]`, while `PC` only advances in the fetch cycle.

- **Fetch cycle (Cycle N)** – bring the LDA instruction into `IR`:
  - `addr = PC`, `EN = 1`, `WEN = 0` → memory outputs the **instruction word** at `M[PC]`.
  - `DATA_MUX` selects memory, `LD_IR = 1` → `out_down/IR` steps to that 32‑bit instruction (for example, `0xABCD0010`).
  - `RED` computes `PC + 4`, with `LD_PC = 1`, `INC_PC = 1` → `PC` moves to the next instruction; `out_down/A` is unchanged.

- **Execute cycle (Cycle N+1)** – load from memory into `A`:
  - `addr = zero_extend(IR[15..0])` → if `IR[15..0] = 0x0010` then `addr = 0x00000010`.
  - `EN = 1`, `WEN = 0` → memory outputs the **data word** at that address (for example, `M[0x00000010] = 0xDEADBEEF`).
  - `DATA_MUX` again selects memory, `A/B_MUX` targets `A`, `LD_A = 1` → `out_down/A` steps to `0xDEADBEEF` while `out_down/IR` stays at the LDA word.

The full, step‑by‑step walkthrough (including all main control signals, the zero‑extenders, and a PlantUML diagram) lives in `LDA_datapath.md` and explains **why the Quartus waveforms look the way they do** for LDA.

## `data_mem` VHDL Design – Detailed Explanation

### Entity Interface

The memory interface is:

- **Clock**: `clk : IN std_logic`  
- **Address**: `addr : IN unsigned(7 DOWNTO 0)`  
  - 8-bit unsigned, allowing addresses `0` to `255` → **256 32-bit words**.
- **Write Data**: `data_in : IN std_logic_vector(31 DOWNTO 0)`  
- **Write Enable**: `wen : IN std_logic`  
  - `'1'` means **write** on the active clock edge.
  - `'0'` means **read**.
- **Enable**: `en : IN std_logic`  
  - `'1'` enables memory access.
  - `'0'` disables memory; `data_out` is forced to zero.
- **Read Data**: `data_out : OUT std_logic_vector(31 DOWNTO 0)`

This is a classic **synchronous RAM**-style interface with separate `en` and `wen` control.

### Internal Memory Representation

Inside the architecture:

- A **type** is defined:
  - `type tmatrix is array(0 to 255) of std_logic_vector(31 downto 0);`
  - Represents the memory array as a VHDL type.
- A **signal** is declared:
  - `signal matrix : tmatrix;`
  - This is the actual memory instance storing the 256 × 32-bit words.

Conceptually, `matrix(addr)` is similar to `mem[addr]` in a software array.

### Clocked Process and Behavior

The behavior is described in a process:

- **Sensitive to the clock**:
  - `process(clk)` with `if falling_edge(clk) then ... end if;`
  - All read/write operations happen on the **falling edge** of `clk`.

- **Enable and Write Enable logic**:
  - When `en = '1'`:
    - If `wen = '0'` → **Read operation**:
      - `data_out <= matrix(to_integer(addr));`
      - The word at `addr` is driven onto `data_out`.
    - If `wen = '1'` → **Write operation**:
      - `matrix(to_integer(addr)) <= data_in;`
      - Stores `data_in` into the memory at the given address.
      - `data_out <= x"00000000";`
      - During a write, the output is forced to zero (instead of showing old or new data).
  - When `en = '0'`:
    - `data_out <= x"00000000";`
    - Memory is effectively **idle**; no reads/writes occur; output is zeroed.

**Key points:**

- All operations are **synchronous** with the clock (no combinational reads).
- The design cleanly separates **read**, **write**, and **disabled** states based on `en` and `wen`.
- Using `to_integer(addr)` converts the `unsigned` `addr` to an integer index into the `matrix` array.

## Concepts Illustrated by This Lab

- **VHDL Entity/Architecture**:
  - `ENTITY` defines the **interface** (ports and types).
  - `ARCHITECTURE` defines the **implementation** (signals, processes, and behavior).

- **Synchronous Memory (RAM)**:
  - Memory operations (read/write) tied to a clock edge (`falling_edge(clk)`).
  - Internal array used to model RAM behavior.
  - Clear separation between **control signals** (`en`, `wen`) and **data signals** (`data_in`, `data_out`).

- **Control Logic for Read/Write**:
  - `en` acts like a **chip enable**.
  - `wen` acts like a **write-enable**; when not asserted, memory performs a **read**.
  - Output behavior is explicitly defined for each mode (read, write, disabled).

- **Hardware-Oriented Types and Libraries**:
  - `ieee.std_logic_1164` provides `std_logic` and `std_logic_vector` for modeling digital signals.
  - `ieee.numeric_std` provides `unsigned` and `to_integer` for arithmetic on vectors.

- **FPGA Design Flow with Quartus**:
  - `.vhd` – source design.
  - `.qpf`, `.qsf`, `.qws` – project and settings.
  - `db`, `incremental_db` – compilation databases.
  - `output_files` – implementation outputs (e.g., programming files).
  - `simulation` + `.vwf` – simulation setup and waveform definitions for verification.

## Overall Summary

This lab builds a **parameterized 32-bit data memory** in VHDL and walks through the typical **FPGA design flow** using Quartus:

- You implement a **synchronous RAM** block with an enable, write-enable, and 8-bit address space using standard VHDL libraries.
- You use Quartus project files to **compile, synthesize, and (optionally) map** the design to a target FPGA.
- You use waveform files and the simulation infrastructure to **verify** correct read/write behavior under different control and timing scenarios.

Together, these files give you a complete example of how a low-level architectural component (data memory) is specified, built, and tested in a real hardware-design toolchain.
