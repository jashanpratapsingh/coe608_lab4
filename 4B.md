## COE 608 Lab 4B – Expected Waveforms for `REG` and `mux_3to1`

Part **4B** extends the memory work of 4A with basic **sequential** and **combinational** building blocks:

- A **register** (`REG`) with **load** and **clear** control.
- A **3-to-1 multiplexer** (`mux_3to1`) that selects one of three 32-bit inputs.

The waveforms for this part demonstrate:

- How a register’s output responds to **LD** (load) and **CLR** (clear).
- How the mux output `y` follows one of the inputs depending on `sel`.

---

## `REG` – Behavior and Waveform Shape

### Interface

From `4b/REG.vhd`:

- **Inputs**:
  - `clk : IN std_logic`
  - `data_in : IN std_logic_vector(31 DOWNTO 0)`
  - `CLR : IN std_logic`
  - `LD : IN std_logic`
- **Output**:
  - `data_out : OUT std_logic_vector(31 DOWNTO 0)`

The process (simplified structurally) is:

- If `LD = '1'`:
  - If `CLR = '1'` → `data_out <= 0x00000000;`
  - If `CLR = '0'` → `data_out <= data_in;`
- Else if `LD = '0'`:
  - If `CLR = '1'` → `data_out <= 0x00000000;`
  - (If `CLR = '0'` and `LD=0`, `data_out` **holds its previous value** – no assignment.)

Note: The code is **not clocked in the usual way** (there is no explicit `if rising_edge(clk)`); instead, the sensitivity list includes `clk, data_in, LD, CLR`, so it acts more like **combinational logic with memory-like behavior**. Functionally for the lab, you can think of it as:

- `LD` is a **write-enable** for the register.
- `CLR` is an **asynchronous clear** that overrides data and forces zero whenever it’s high.

### Expected Waveform Scenarios

1. **Initial / Idle State**
   - `LD = 0`, `CLR = 0`.
   - `data_out` retains its last value (could be `X` at time 0 until first assignment).
   - As `data_in` toggles while `LD=0` and `CLR=0`, **`data_out` stays flat**.

2. **Clear While Idle**
   - `LD = 0`, `CLR` goes from `0` to `1`.
   - Process executes and takes the `LD='0'` branch:
     - `CLR='1'` → `data_out <= 0x00000000`.
   - **Waveform**:
     - At the moment `CLR` rises to `1`, you see `data_out` drop to **0x00000000** and remain there as long as `CLR=1`.

3. **Load New Data**
   - `LD` transitions from `0` to `1` while `CLR = 0` and `data_in` contains some non-zero word.
   - Process takes the `LD='1'` and `CLR='0'` branch:
     - `data_out <= data_in;`
   - **Waveform**:
     - `data_in` is stable at, say, `0xDEADBEEF`.
     - At the instant `LD` goes high (with `CLR=0`), `data_out` steps from its old value (possibly `0`) to `0xDEADBEEF` and remains at that value even if `data_in` changes later while `LD` returns to `0`.

4. **Load with Clear Active**
   - `LD = 1`, `CLR` toggles from `0` to `1`:
     - First branch tested is `LD='1'`, then inside:
       - If `CLR='1'` → `data_out <= 0x00000000;`
   - **Waveform**:
     - Regardless of `data_in`, as soon as `CLR`=1 with `LD=1`, `data_out` is driven to **0**.
     - When `CLR` goes back to `0` while `LD`=1, `data_out` will **immediately track `data_in`** (since the LD=1 branch with CLR=0 assigns `data_in`).

5. **Hold Behavior**
   - Once `LD` returns to `0` and `CLR=0`, there are **no assignments** to `data_out`.
   - The register “remembers” its last driven value.
   - On the waveform:
     - `data_out` becomes flat (no further steps) while `LD=0` and `CLR=0`, even if `clk` and `data_in` toggle.

Overall, the `REG` waveforms demonstrate:

- **Asynchronous clear** behavior: `CLR` can force the output low at any time.
- **Load gating**: `LD` chooses whether `data_in` updates the register or the old value is held.

## `mux_3to1` – Behavior and Waveform Shape

### Interface

From `4b/mux3to1.vhd`:

- **Inputs**:
  - `in0, in1, in2 : IN std_logic_vector(31 DOWNTO 0)`
  - `sel : IN std_logic_vector(1 DOWNTO 0)`
- **Output**:
  - `y : OUT std_logic_vector(31 DOWNTO 0)`

Behavior:

- If `sel = "00"` → `y <= in0;`
- If `sel = "01"` → `y <= in1;`
- If `sel = "10"` → `y <= in2;`
- Otherwise → `y <= "ZZZZ...Z"` (32 bits of high-impedance `'Z'`).

The process is sensitive to `sel`, `in0`, `in1`, `in2`, so it behaves as a **pure combinational mux**.

### Expected Waveform Scenarios

Assume we drive three distinct constant words on the inputs:

- `in0 = 0x00000001`
- `in1 = 0x00000002`
- `in2 = 0x00000003`

Then change `sel` over time:

1. **Select in0 (`sel="00"`)**
   - `sel` is `"00"`; process assigns `y <= in0;`.
   - **Waveform**:
     - `y` shows `0x00000001`, following `in0`.
     - If `in0` toggles while `sel="00"`, `y` toggles in lockstep.

2. **Select in1 (`sel="01"`)**
   - At the instant `sel` changes to `"01"`, the mux changes branch:
     - `y <= in1;`
   - **Waveform**:
     - `y` steps from `0x00000001` to `0x00000002`.
     - It continues to follow `in1` until `sel` changes again.

3. **Select in2 (`sel="10"`)**
   - When `sel = "10"`:
     - `y <= in2;`
   - **Waveform**:
     - `y` steps to `0x00000003` and tracks `in2`.

4. **Invalid selector (`sel="11"` or others)**
   - The `when others` branch:
     - `y <= "ZZZZ....Z";`
   - **Waveform**:
     - In a digital waveform viewer, `'Z'` may appear as high-impedance (often drawn as a baseline with a label `Z`).
     - This indicates that the mux is not actively driving a logic `0`/`1` pattern.

Because this is combinational logic:

- There is **no clock** involved in this decision.
- `y` responds **immediately** to any change in `sel` or the selected input, apart from tiny simulation delta delays.
- The waveform for `y` is a sequence of **flat segments** corresponding to whichever input is selected at that time, with step changes exactly when `sel` or the active input changes.

## Combined Intuition for 4B Waveforms

In a typical lab testbench for part 4B, you would:

- Use the **mux** to select between:
  - A constant, a previously computed value, or a memory read.
- Then **feed the mux output into `data_in` of `REG`**.
- Use `LD` and `CLR` to control when the register captures the mux result.

On a combined waveform:

- `sel` determines which of the inputs is appearing on the mux output `y`.
- At times when `LD=1` and `CLR=0`, the **current value on `y`** is sampled and becomes the new `data_out` of the register.
- When `CLR=1`, regardless of `sel` and `LD`, the register output is driven to **0**.
- When `LD=0` and `CLR=0`, `data_out` stays constant, even if `sel` keeps switching and `y` is changing.

So the **visual pattern** is:

- `y` quickly tracks changes in `sel` and its inputs.
- `data_out` changes only when **LD or CLR events** occur, forming well-defined steps that represent the register capturing (or clearing) values originating from the mux.
