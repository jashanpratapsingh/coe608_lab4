## COE 608 Lab 4A – Expected Waveforms for `data_mem`

This describes how the **waveforms should look** for part **4A** (the `data_mem` VHDL module) and **why** the signals behave that way. It is based on the VHDL in `data_mem.vhd` / `4a.vhd` and the Quartus waveform file `data_mem.vwf`.

---

## Key Signals

- **`clk`**: clock input (single bit).
- **`en`**: memory enable.
- **`wen`**: write enable (with `en = '1'`):
  - `wen = '1'` → **write**.
  - `wen = '0'` → **read**.
- **`addr[7:0]`**: 8-bit unsigned address bus, selects one of 256 memory locations.
- **`data_in[31:0]`**: 32-bit data written into memory when `wen = '1'`.
- **`data_out[31:0]`**: 32-bit data read from memory, or zero when disabled or during writes.

The architecture uses:

- A `process(clk)` with `if falling_edge(clk)` so all operations are synchronous on the **falling edge** of `clk`.
- An internal 256×32 array (`matrix`) to store the memory contents.

---

## Clock and Time Base

From `data_mem.vwf`:

- **Time unit**: ns.
- **Data duration**: **340 ns**.
- **Clock**: inside `TRANSITION_LIST("clk")`:
  - Repeats `LEVEL 0 FOR 20 ns` then `LEVEL 1 FOR 20 ns`, 8 times, then `LEVEL 0 FOR 20 ns`.
  - So `clk` is:
    - Period: 40 ns (20 ns low + 20 ns high).
    - Frequency: 25 MHz (not critical, but good to know).
    - There are **8 full cycles** in the 0–320 ns window, plus a final low period.
  - **Falling edges** happen at: 20 ns, 60 ns, 100 ns, 140 ns, 180 ns, 220 ns, 260 ns, 300 ns.

On the waveform:

- You’ll see a **square wave** for `clk` toggling every 20 ns.
- Memory **state changes** (reads/writes applied to `matrix` and `data_out`) should occur exactly at these **falling edges**.

---

## `en` and `wen` Control Patterns

From the waveform transitions:

- **`en`**:
  - 0–20 ns: `0`
  - 20–60 ns: `1`
  - 60–140 ns: `0`
  - 140–180 ns: `1`
  - 180–220 ns: `0`
  - 220–250 ns: `1`
  - 250–260 ns: `0`
  - 260–300 ns: `1`
  - 300–340 ns: `0`

- **`wen`**:
  - 0–20 ns: `0`
  - 20–60 ns: `1`
  - 60–100 ns: `0`
  - 100–130 ns: `1`
  - 130–220 ns: `0`
  - 220–250 ns: `1`
  - 250–340 ns: `0`

Interpretation over **falling edges**:

- At each falling edge, the combination of `(en, wen)` determines what happens:

| Falling edge (approx.) | `en` | `wen` | Operation                                  | Expected behavior on `data_out`             |
|------------------------|------|-------|--------------------------------------------|---------------------------------------------|
| 20 ns                  | 1    | 1     | **Write** first word to selected address   | `data_out` forced to `0x00000000`           |
| 60 ns                  | 0    | 0     | **Disabled**                               | `data_out` forced to `0x00000000`           |
| 100 ns                 | 0    | 0     | **Disabled**                               | `data_out` forced to `0x00000000`           |
| 140 ns                 | 1    | 1     | **Write** second word                      | `data_out` forced to `0x00000000`           |
| 180 ns                 | 0    | 0     | **Disabled**                               | `data_out` forced to `0x00000000`           |
| 220 ns                 | 1    | 0     | **Read** (uses address and previous writes)| `data_out` becomes the stored word          |
| 260 ns                 | 1    | 0     | **Read**                                   | `data_out` corresponds to current address   |
| 300 ns                 | 1    | 0     | **Read**                                   | `data_out` corresponds to current address   |

So the **waveform shape** of `data_out` will be:

- Initially `X` (unknown) until the first clock events.
- Forced to `0x00000000` during writes and when `en = '0'`.
- Later, when `en = '1'` and `wen = '0'` at a falling edge, `data_out` **jumps from zero/unknown to a specific 32-bit value** that was previously written to `matrix` at that address.

---

## Address (`addr`) and Data-In (`data_in`) Patterns

In the `.vwf` file:

- `addr[7]` changes from `0` to `1` after 220 ns; others largely stay at `0`, except `addr[3]` which toggles:
  - `addr[3]`:
    - 0–10 ns: `0`
    - 10–220 ns: `1`
    - 220–340 ns: `0`
- This means for most of the simulation:
  - The address is something like **0b00001000** (if only bit 3 is `1`) or **0b10001000** after 220 ns (when bit 7 becomes `1`).
- `data_in` bits define one or two distinct **32-bit patterns**:
  - Some bits change from `0` to `1` at around 220 ns (like bit 31, bit 18, bit 22, bit 14, bit 5).
  - Combined, they form **two different data words**:
    - **First write**: when `en=1, wen=1` around 20 ns (at the initial address) – this stores a first pattern.
    - **Second write**: when `en=1, wen=1` around 140 ns – this stores another pattern (possibly at the same or a different address, depending on `addr`).

On the waveform:

- You’ll see `addr` and `data_in` are **set up and stable** **before** the falling edges where writes occur.
- At write cycles:
  - `data_in` shows a specific hex value in the hex view.
  - `data_out` is held at zero (per design) even though the internal `matrix` is updated.
- At read cycles:
  - `addr` is again stable before the falling edge.
  - After that edge, `data_out` jumps to the **stored value** corresponding to the current address.

This matches the behavioral code:

- `if en = '1' then`
  - `if wen = '0'` → `data_out <= matrix(to_integer(addr));`
  - `elsif wen = '1'` → `matrix(to_integer(addr)) <= data_in; data_out <= x"00000000";`
- `elsif en = '0'` → `data_out <= x"00000000";`

## Expected `data_out` Waveform Shape

Putting it all together, the **ideal `data_out` waveform** for part 4A looks like:

1. **0–20 ns**:
   - `en = 0`, `wen = 0`, clock toggling.
   - At the 20 ns falling edge, no effective read or write (since `en=0`).
   - `data_out` stays at `0` or `X`, but by design it is driven to `0` when `en=0`.

2. **20–60 ns (First write window)**:
   - `en = 1`, `wen = 1`.
   - At the 20 ns falling edge (already occurred at boundary), and subsequent falling edges before `en`/`wen` change:
     - The selected address gets written with `data_in`.
     - `data_out` is forced to `0x00000000`.
   - Visually: `data_out` is flat at **0**.

3. **60–140 ns (Disabled / second write)**:
   - `en = 0` until 140 ns → `data_out` is held at zero.
   - `en = 1`, `wen = 1` around 140 ns:
     - At that falling edge, another write occurs.
     - `data_out` remains **0** during the write, no visible non-zero values yet.

4. **180–220 ns (Disabled again)**:
   - `en = 0` → `data_out` stays forced to **0**.

5. **220–340 ns (Read window)**:
   - `en = 1`, `wen = 0` for most falling edges.
   - At 220 ns, 260 ns, 300 ns falling edges:
     - `data_out` updates to the stored contents of `matrix` at the current `addr`.
     - You’ll see **step changes** in `data_out`:
       - Flat zero before the read.
       - Jump to first stored value at the first read edge.
       - Possibly jump again if `addr` changes to a different location.

So the final waveform for `data_out` over time:

- Mostly **0** when:
  - `en = 0` (disabled), or
  - `en = 1` & `wen = 1` (write cycles).
- Distinct **non-zero hex words** during:
  - `en = 1` & `wen = 0` after a falling edge, reflecting stored memory contents.

---

## Why the Waveform Looks This Way

- The VHDL is a **synchronous memory**:
  - Operations only take effect on **clock edges** (here, falling edges).
  - This is why `data_out` changes only at those discrete times, not mid-cycle.
- The **separate `en` and `wen`** allow:
  - Completely disabling the memory (output zero).
  - Selecting between **read** and **write** when enabled.
- During a **write**, the design explicitly chooses to **mask the output** to zero (`data_out <= 0`), rather than reflecting old or new data:
  - This simplifies the datapath interface and avoids ambiguous “read-while-write” behavior.
- The **waveform file** is crafted to move through all these scenarios:
  - Write one pattern, write another, then read them back at later times and/or addresses.

In summary, the 4A waveform should clearly show:

- Regular clock edges.
- Control windows where writes/reads occur (driven by `en` and `wen`).
- `data_out` staying zero when disabled or writing, and taking on stored values only when a read is requested at a falling clock edge.
